
# Edgeloop Architecture Plan

## Current Stack (2026)
- **Framework**: Next.js 16 (App Router with Partial Prerendering)
- **Runtime**: Node.js 20+ / Bun 1.2+
- **Backend**: Express server with TypeScript
- **Database**: PostgreSQL with Drizzle ORM
- **State**: Zustand for client state management
- **Styling**: Tailwind CSS v4
- **Analytics**: Python engine (NumPy, Pandas, scikit-learn)

## Architecture Overview

### Structure
```
edgeloop-1/
├── app/                    # Next.js 16 App Router
│   ├── components/         # Shared React components
│   ├── api/               # API routes
│   └── [routes]/          # Page routes
├── server/                # Express backend
│   ├── services/          # Business logic & data services
│   ├── infrastructure/    # Core infrastructure (logging, caching, etc.)
│   ├── analytics/         # Analytics services
│   └── betting/           # Betting-specific services
├── python_engine/         # Statistical models & ML
├── shared/                # Shared types & schemas
└── requirements/          # Business requirements documentation
```

### Data Flow
1. `shared/` - Type definitions and schemas (source of truth)
2. `server/services/*` - Business logic and data aggregation
3. `server/routes.ts` - API endpoint definitions
4. `app/api/*` - Next.js API routes (proxy to Express)
5. `app/components/*` - React components with data fetching

## Code Quality Priorities

### [CRITICAL]
1. **Type Safety**: Eliminate `any` types in core server logic
   - `server/index.ts` - Server lifecycle and error handling
   - `server/db.ts` - Database connection management
   - `server/services/statsAggregator.ts` - Data normalization

2. **Error Handling**: Implement structured error handling
   - Use `server/infrastructure/logger.ts` consistently
   - Ensure error handlers don't throw after response sent
   - Add proper error boundaries in React components

3. **Database Safety**: 
   - Make `db` non-nullable with typed guards
   - Implement connection pooling properly
   - Add transaction support where needed

4. **API Contracts**: 
   - Align types between server routes and client API calls
   - Use Zod schemas for runtime validation
   - Document API endpoints with OpenAPI/Swagger

### [IMPORTANT]
1. **Logging**: Replace all console.log with structured logger
2. **Documentation**: Add JSDoc for complex functions
3. **Code Organization**: Remove dead code and unused imports
4. **Testing**: Maintain test coverage for critical paths

### [MAINTENANCE]
1. **Formatting**: Consistent code style (Prettier)
2. **Naming**: Follow TypeScript naming conventions
3. **Comments**: Remove redundant comments, keep meaningful ones

## Development Workflow

### Type Safety
- Primary source: `shared/schema.ts` and `shared/models/*`
- Server DTOs: Align with shared types
- Client types: Import from shared, avoid duplication

### Testing Strategy
```bash
npm run check      # TypeScript type checking
npm run lint       # ESLint validation
npm test           # Run test suite
npm run test:coverage  # Coverage report
```

### Performance Optimization
- Use React Server Components where possible
- Implement proper caching strategies (Redis)
- Optimize database queries with indexes
- Use connection pooling for external APIs

## Infrastructure

### Caching Strategy
- Redis for frequently accessed data
- In-memory cache for static data
- Cache invalidation on data updates

### Error Recovery
- Circuit breakers for external services
- Automatic retry with exponential backoff
- Graceful degradation when services fail

### Monitoring
- Structured logging with correlation IDs
- Performance metrics tracking
- Error rate monitoring
- API latency tracking

## Deployment

### Environments

1. Type checking (`npm run check`)
